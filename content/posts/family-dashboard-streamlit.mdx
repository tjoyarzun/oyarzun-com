I built us a private family dashboard six months ago on a whim during a long weekend. It has since become the thing I check more than my work dashboards. Here is the whole thing, step by step.

## What It Tracks

- **Activity:** Strava hikes, runs, bike rides (both of us)
- **Reading:** Goodreads currently-reading and read shelves
- **Kitchen:** A SQLite database of recipes we have made and rated
- **Upcoming:** A simple events table I maintain manually
- **Stats:** Running totals for the year (miles, books, ski days, recipes tried)

The goal was not comprehensiveness — it was to replace the five different apps and spreadsheets I would otherwise check separately.

## Architecture

```
Strava API ──────────┐
Goodreads RSS ────────┼──► Python ETL ──► SQLite ──► Streamlit
Recipe CSV uploads ───┘
```

Everything lives on a small DigitalOcean droplet ($6/month). The ETL runs via cron every four hours. Streamlit is served behind a Caddy reverse proxy with basic auth — nothing fancy, just enough to keep it private.

## The Strava Integration

Strava's API requires OAuth, which is the most annoying part of the whole setup. Once you get through that, the data is excellent.

```python
import requests

def get_strava_activities(access_token: str, per_page: int = 50) -> list[dict]:
    resp = requests.get(
        "https://www.strava.com/api/v3/athlete/activities",
        headers={"Authorization": f"Bearer {access_token}"},
        params={"per_page": per_page}
    )
    resp.raise_for_status()
    return resp.json()
```

I store the raw JSON in a `raw_activities` table and transform it into a clean `fct_activities` table with normalized fields. This way I can re-run the transformation without hitting the API again.

The token refresh logic is the part that bites everyone:

```python
def refresh_strava_token(client_id, client_secret, refresh_token):
    resp = requests.post("https://www.strava.com/oauth/token", data={
        "client_id": client_id,
        "client_secret": client_secret,
        "refresh_token": refresh_token,
        "grant_type": "refresh_token"
    })
    data = resp.json()
    # Store new refresh_token — it rotates on every refresh
    save_refresh_token(data["refresh_token"])
```

## The Goodreads Hack

Goodreads does not have a real API anymore (RIP). But they still publish RSS feeds per shelf. This is technically undocumented but has worked reliably for two years:

```
https://www.goodreads.com/review/list_rss/{user_id}?shelf=currently-reading
https://www.goodreads.com/review/list_rss/{user_id}?shelf=read
```

Parse with `feedparser`, extract the title/author/rating, done.

## What I Would Do Differently

**Use Parquet instead of SQLite.** SQLite is fine, but we are already using DuckDB for the main family data stack (see Tommy's post). I would consolidate everything there.

**Add a mobile view.** Streamlit's mobile experience is mediocre. If I rebuild this, I would use Next.js (ironic given this site) or just accept the desktop-only constraint.

**Automate the recipe input.** Right now I upload a CSV every time we try a new recipe. A simple form would be better. Maybe a Google Form → Google Sheets → Python pipeline.

Despite those rough edges, this dashboard has genuinely changed how we talk about our weekends. Highly recommend building something like it, even if your version is way less elaborate.
